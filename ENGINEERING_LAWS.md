# Engineering Laws

Фундаментальные принципы, по которым строится и развивается проект.

Эти законы обязательны при проектировании, рефакторинге и добавлении нового функционала.

## Статус документа

`ENGINEERING_LAWS.md` — главный нормативный документ проекта.

Если по одному и тому же вопросу существуют несколько рабочих документов, приоритет такой:

1. `ENGINEERING_LAWS.md` — обязательные инженерные законы.
2. `ARCHITECTURE.md` — фактическая карта текущего устройства проекта.
3. `AI_COLLABORATION_RULES.md` — короткая производная версия правил для рабочих ИИ-сессий.
4. `PROJECT_RULES.md` — переходный файл совместимости, не главный источник правил.

---

## 1. Закон разделения ответственности

Каждый модуль должен отвечать только за одну зону:

- UI → отображение
- сервисы → сценарии и orchestration
- расчеты → чистая бизнес-логика
- репозитории → доступ к данным

Запрещено смешивать эти зоны в одном месте.

---

## 2. Закон слабой связности

Модули должны минимально зависеть друг от друга.

Изменение одного модуля не должно требовать переписывания других.

Если изменение тянет за собой цепочку правок — архитектура нарушена.

---

## 3. Закон локальности изменений

Любое изменение должно быть максимально локальным.

Если для добавления функции нужно менять много несвязанных частей системы — структура выбрана неправильно.

---

## 4. Закон чистых доменных правил

Ключевые правила системы (расчеты, логика заявок, статусы) не должны зависеть от:

- Telegram
- React UI
- формата отображения

Доменные правила должны быть изолированы и переиспользуемы.

---

## 5. Закон независимости интерфейса

UI можно менять без изменения бизнес-логики.

Изменение UX не должно требовать переписывания расчетов или сервисов.

---

## 6. Закон явных границ

Проект должен иметь четкие слои:

- presentation (Telegram, API, React)
- application (сценарии)
- domain (расчеты и правила)
- infrastructure (БД, внешние сервисы)

Нельзя размывать границы между слоями.

---

## 7. Закон явных контрактов

Модули взаимодействуют через понятные структуры данных.

Запрещено:
- передавать “сырые” dict без структуры
- завязываться на случайные поля

Каждая важная сущность должна иметь модель.

---

## 8. Закон композиции

Система строится из модулей, а не из монолитов.

Любой крупный файл или компонент должен быть разбит на части по ответственности.

---

## 9. Закон предсказуемого места

У каждой новой функции должно быть очевидное место в проекте.

Если непонятно, куда добавить код — архитектура нарушена.

---

## 10. Закон дешевых изменений

Главная цель архитектуры — удешевить изменения.

Перед добавлением кода всегда проверяется:

- можно ли будет легко изменить это позже?
- не усложнит ли это развитие системы?

---

## 11. Закон минимальной достаточности

Не добавлять архитектуру “на будущее”, если она не решает текущую задачу.

Но и не оставлять решение, которое очевидно сломается при росте.

---

## 12. Закон контроля роста

Любая точка системы, которая начинает разрастаться (например `storage.py`, `dialogue.py`, `calculator.tsx`), должна быть разделена.

Рост без декомпозиции запрещен.
