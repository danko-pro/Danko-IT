# PROJECT_RULES.md

## 0. Общий принцип

Любое изменение в проекте выполняется по схеме:

**архитектурное размещение -> проверка -> реализация -> краткое объяснение**

Нельзя сразу писать код без понимания, где он должен жить.

## 1. Архитектурная модель проекта

Проект разделён на слои:

- **transport (routers, handlers)**
  Принимает вход (Telegram, HTTP), передаёт дальше

- **services (бизнес-логика)**
  Основные действия, сценарии, процессы

- **clients / integrations**
  Внешние API (AI, платежи, сторонние сервисы)

- **core / infrastructure**
  Инициализация, конфиг, middleware, логирование

- **plugins / features**
  Отдельные независимые функции

- **schemas / dto**
  Структуры данных

## 2. Главные запреты

Запрещено:

- писать бизнес-логику в handlers
- вызывать внешние API напрямую из router
- дублировать функции в разных файлах
- создавать "временные" решения без структуры
- смешивать слои (например: router + логика + API в одном месте)
- размазывать конфигурацию по проекту
- писать код "на будущее" без необходимости

## 3. Правило размещения кода

Перед написанием кода ОБЯЗАТЕЛЬНО определить:

1. К какому слою относится задача
2. В какой файл она должна попасть
3. Нужно ли создавать новый модуль
4. Не существует ли уже похожей логики

Если место не определено - код писать нельзя.

## 4. Формат работы с задачей

Каждое изменение должно идти в таком порядке:

### Шаг 1. Архитектурное решение

- куда идёт код
- почему именно туда

### Шаг 2. Проверка

- нет ли дублирования
- не ломает ли структуру
- не перегружает ли слой

### Шаг 3. Код

- чистый
- минимальный
- без лишнего

### Шаг 4. Объяснение

- кратко: что сделано и почему так

## 5. Стиль кода

- функции короткие
- одна функция - одна задача
- имена говорящие
- без магии
- без скрытых зависимостей
- без "универсальных костылей"

## 6. Комментарии (ОЧЕНЬ ВАЖНО)

Комментарии должны:

- объяснять **смысл**, а не синтаксис
- быть короткими
- использоваться только там, где нужно
- не дублировать очевидное

### Хорошо:

```python
# Отправляем запрос к AI и обрабатываем возможные ошибки сети
```

### Плохо:

```python
# вызываем функцию
response = func()
```

### Обязательно:

- docstring для ключевых функций и классов
- комментарии на сложных местах
- комментарии на архитектурных точках

### Запрещено:

- комментировать каждую строку
- писать "водяные" комментарии
- превращать код в текстовую помойку

## 7. Работа с внешними API

Правила:

- только через отдельный client / manager
- обязательны:
  - таймауты
  - обработка ошибок
  - fallback
- нельзя вызывать API из разных мест напрямую

## 8. Handler (router) правила

Handler должен:

- принять данные
- провалидировать
- вызвать сервис
- вернуть результат

Handler НЕ должен:

- содержать бизнес-логику
- работать с API напрямую
- делать сложные вычисления

## 9. Масштабирование

При добавлении нового функционала:

- сначала ищется существующий слой
- если нет - создаётся новый модуль, но по правилам
- нельзя "впихнуть" код в ближайший файл

## 10. Анти-хаос контроль

Перед финальным кодом проверять:

- не стал ли файл слишком большим
- не появилось ли дублирование
- не нарушена ли архитектура
- не добавлен ли временный костыль

Если хотя бы один пункт "да" -> решение переделывается

## 11. Обязательные проверки перед ответом

Перед тем как выдать код, всегда проверить:

- где будет жить код
- соответствует ли слой
- нет ли дублирования
- не нарушена ли структура
- понятен ли код без лишних комментариев

## 12. Принцип "чистого роста"

Каждое изменение должно:

- улучшать читаемость
- сохранять структуру
- не усложнять проект
- не создавать долг

## 13. Формат ответа (строго)

Каждый ответ должен содержать:

1. Что делаем
2. Куда это ложится по архитектуре
3. Код
4. Почему это правильное решение

## 14. Ключевая установка

**Сначала структура - потом код.**

Если это правило нарушено -> решение считается неправильным.

## 15. Финальная формула

Хорошее решение:

- лежит в правильном месте
- не ломает архитектуру
- не дублирует код
- понятно читается
- минимально по объёму

Плохое решение:

- "работает, но хз где лежит"
- добавляет хаос
- смешивает слои
- требует объяснений после написания

## 16. Главное правило проекта

**Код должен быть таким, чтобы через 3 месяца его понял другой разработчик без объяснений.**

## Как этим пользоваться

### В работе с GPT

Каждый раз просто добавляешь:

> Работай строго по PROJECT_RULES.md

### В проекте

Положи файл в корень репозитория.

### При любой задаче

Добавляешь одну фразу:

> Сначала архитектурное размещение, потом код.

## Что это тебе даст

- не будет разрастания handler-ов
- не будет "где попало" функций
- не будет дублирования
- не будет каши при масштабировании
- код станет предсказуемым
