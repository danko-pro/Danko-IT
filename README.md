# Supply Bot

Telegram-бот, backend админки и React-интерфейс для:
- групповых заявок на материалы;
- каталога материалов;
- проектного калькулятора;
- расчётов, спецификаций и смет.

## Документы проекта

- `ENGINEERING_LAWS.md` — главный инженерный документ проекта.
- `ARCHITECTURE.md` — карта текущей архитектуры и ближайшего плана развития.
- `AI_COLLABORATION_RULES.md` — короткие правила работы ИИ с репозиторием.
- `PROJECT_RULES.md` — переходный compatibility-документ.

## Что уже есть

- загрузка конфигурации из `.env.local`, `.env` или `settings.env`;
- SQLite-хранилище для каталога, заявок и проектного калькулятора;
- Telegram-админка в личке:
  - семейства;
  - варианты;
  - SKU;
  - алиасы;
  - неизвестные термины;
  - поиск;
  - настройки окна доставки;
- групповой сценарий заявок:
  - распознавание материалов;
  - уточнение обязательных полей;
  - проверка времени доставки;
  - подтверждение заявки;
  - отправка администратору;
- локальная web-админка на `FastAPI + React`;
- калькулятор по проекту, комнатам, дверям, тёплому полу, напольным покрытиям и отделке стен.

## Конфигурация

По умолчанию проект ищет конфиг в таком порядке:

1. `.env.local`
2. `.env`
3. `settings.local.env`
4. `settings.env`

Рекомендуемая схема:
- `.env.example` держать как безопасный шаблон;
- реальные ключи хранить в `.env.local`;
- при необходимости использовать `SUPPLY_BOT_CONFIG` для явного пути к приватному конфигу.

Важно:
- реальные ключи не должны храниться в репозитории;
- `BOT_TOKEN`, `OPENAI_API_KEY`, `MISTRAL_API_KEY` и другие секреты держите только в локальных файлах;
- файл `.env.local` уже игнорируется через `.gitignore`.

## Запуск бота

1. Установить зависимости:

```powershell
pip install -r requirements.txt
```

2. Подготовить локальный конфиг, обычно `.env.local`.

3. Запустить бота:

```powershell
python run_bot.py
```

## Базовые проверки кода

Для локальной инженерной гигиены есть отдельный dev-набор:

```powershell
pip install -r requirements-dev.txt
```

Минимальный baseline сейчас такой:

```powershell
python -m ruff check src tests --no-cache
python -m unittest discover -s tests -v
```

Важно:
- `ruff` пока используется мягко, без массовой переформатировки всего проекта;
- текущий backlog линтера в основном состоит из `E501` по длинным строкам;
- новые и уже перерабатываемые модули стоит подтягивать к baseline постепенно.

## Локальная админка

Backend:

```powershell
python run_admin.py
```

API поднимается на:

```text
http://127.0.0.1:8000
```

Проверка здоровья:

```text
http://127.0.0.1:8000/api/health
```

Frontend:

```powershell
cd admin-ui
npm install
npm run dev
```

UI открывается на:

```text
http://127.0.0.1:5173
```

Если нужен другой адрес API:

```powershell
$env:VITE_API_BASE_URL="http://127.0.0.1:8000"
```

## Важно для Telegram

- Для группового сценария у бота должен быть доступ ко всем сообщениям в группе.
- Для этого у Telegram-бота нужно отключить privacy mode через `BotFather`.

## LLM-режим

Если ключи LLM не заданы, бот всё равно работает, но отвечает без нейросетевой перефразировки.

Для официального OpenAI:

- `OPENAI_API_KEY=<ключ>`
- `OPENAI_BASE_URL=https://api.openai.com/v1`
- `SUPPLY_DIALOGUE_PRIMARY_PROVIDER=openai`
- `SUPPLY_DIALOGUE_MODEL=gpt-5.4`

## Архитектурный ориентир

- диалоговый слой остаётся `LLM-first`;
- код отвечает за применение действий, валидацию, ограничения и fallback;
- расчёты, storage, admin API и frontend должны развиваться как отдельные подсистемы, а не как новые монолиты.
