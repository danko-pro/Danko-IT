# Contract Intelligence

## Назначение

`Contract Intelligence` — это не просто блок с реквизитами договора.
Это управляемый раздел карточки объекта, который помогает менеджеру:

- видеть ключевые условия договора;
- понимать, что было извлечено из документа нейросетью;
- не пропускать даты выставления счетов;
- не пропускать ожидаемые оплаты и договорные вехи;
- быстро подтверждать, что действие уже выполнено.

На текущем этапе блок живет внутри `dashboard` как UI-прототип карточки объекта.
В целевой архитектуре он должен стать частью отдельного домена `projects`.

## Что блок должен делать

Блок договора отвечает за 4 слоя:

1. `Document`
- загруженный файл договора;
- оригинальный PDF/скан/фото;
- имя файла и технические метаданные.

2. `Extraction`
- распознанные нейросетью поля;
- статус извлечения;
- статус ручной проверки;
- источник правды до подтверждения пользователя не считается окончательным.

3. `Milestones`
- график выставления счетов;
- график ожидаемых оплат;
- договорные вехи;
- срок окончания этапов и договора.

4. `Alerts`
- ближайшее действие;
- просроченное действие;
- upcoming-событие;
- предупреждения о критичных датах.

## Что нейросеть должна извлекать

Минимальный обязательный набор:

- название договора;
- номер договора;
- дата подписания;
- дата начала;
- предполагаемая дата окончания;
- сумма договора;
- условия первого аванса;
- правила последующих оплат;
- список платежных вех;
- список ключевых этапов / дедлайнов.

Желательно извлекать позже:

- штрафные условия;
- обязательные документы для следующей оплаты;
- зависимость оплаты от акта, счета или этапа;
- специальные условия переноса сроков.

## Что менеджер должен видеть в интерфейсе

В блоке договора должны быть следующие зоны:

1. `Hero`
- имя файла;
- название договора;
- номер;
- статус `AI требует проверки` / `Проверено`.

2. `Ключевые даты и сумма`
- подписан;
- старт;
- план завершения;
- сумма договора.

3. `Условия оплаты`
- короткий текстовый summary условий аванса и следующих платежей.

4. `Ближайшее действие`
- что именно нужно сделать;
- до какой даты;
- если есть сумма — показать сумму;
- CTA-кнопка подтверждения действия.

5. `График вех`
- список всех ближайших и закрытых milestones;
- дата;
- сумма, если есть;
- статус;
- короткая заметка.

## Интерактивность

MVP должен поддерживать:

- отметить ближайшую веху выполненной;
- сразу показать следующую активную веху;
- визуально отличать `due`, `upcoming`, `completed`;
- дать менеджеру понять, что сейчас требует внимания.

Следующий слой после MVP:

- ручное редактирование извлеченных полей;
- загрузка нового договора;
- перепривязка milestones;
- комментарий менеджера;
- подтверждение, что AI-извлечение проверено.

## Ограничения

- блок договора не должен сам владеть финансовой аналитикой объекта целиком;
- блок договора не должен считать смету;
- блок договора не заменяет логистику, а дает ей даты и milestone-события;
- AI extraction не должен silently перезаписывать ручные правки менеджера;
- все критичные даты должны быть подтверждаемыми и редактируемыми вручную.

## Минимальная модель данных

```ts
type ProjectContract = {
  id: string;
  fileName: string;
  title: string;
  number: string;
  signedAt: string | null;
  startDate: string | null;
  plannedEndDate: string | null;
  amount: number | null;
  advanceTerms: string;
  extractionStatus: "review" | "verified";
  milestones: ProjectContractMilestone[];
};

type ProjectContractMilestone = {
  id: string;
  kind: "invoice" | "payment" | "deadline";
  title: string;
  plannedDate: string;
  amount?: number;
  note?: string;
  status: "upcoming" | "due" | "completed";
};
```

## Источники данных

Сейчас:

- мок-данные внутри `dashboard`.

Дальше:

- калькулятор объекта дает базовую сумму и контекст проекта;
- договор загружается пользователем;
- AI extraction извлекает поля и milestones;
- менеджер подтверждает или исправляет результат;
- backend хранит уже нормализованную модель договора и вех.

## Порядок реализации

1. Сделать устойчивую domain-модель договора и milestones.
2. Поднять backend-схему хранения договора.
3. Сделать upload + временный parse-result mock.
4. Сделать экран ручной проверки извлеченных данных.
5. Подключить AI extraction.
6. Подключить alerts и ближайшее действие.
7. Отдать milestone-даты в логистику и общую карточку проекта.

## Чеклист, чтобы ничего не забыть

- сохранять оригинальный файл договора;
- хранить extracted fields отдельно от user-confirmed fields, если потребуется;
- не терять сумму и даты при повторной загрузке версии договора;
- учитывать несколько оплат и несколько этапов;
- показывать ближайшую веху отдельно от полного списка;
- не смешивать UI договора с общей бухгалтерией объекта;
- предусмотреть ручную корректировку любого распознанного поля;
- уметь отметить milestone выполненной;
- использовать один источник правды для contract milestones и alerts.
