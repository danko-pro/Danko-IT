# Danko-IT / Danko BuildTech — roadmap запуска бота, AI и калькулятора

Дата фиксации: 2026-05-24

## 1. Текущий контекст

Danko-IT / Danko BuildTech развивается как отделочная IT-компания:
- публичный сайт продаёт ремонт квартир и апартаментов под ключ;
- админка живёт отдельно на `/app`;
- дальше система должна принимать заявки, передавать их в бот, помогать структурировать данные через AI и постепенно заменить Google Sheets-калькуляторы веб-калькулятором.

Сейчас официальный домен ещё не запущен, а сайт не считается готовым к полноценному публичному продвижению.

Поэтому приоритет меняется:
сначала не домен и не полировка лендинга, а инфраструктура заявки, бот, AI и калькулятор.

## 2. Главный приоритет

До официального запуска домена нужно закрыть три больших направления:

1. Запустить канал заявок через бота.
2. Подключить AI как отдельный backend-сервис, без смешивания с роутерами и UI.
3. Начать перенос калькулятора из Google Sheets в сайт.

Лендинг, визитка, QR и домен остаются важными, но ниже по приоритету до момента, пока система не сможет принимать и обрабатывать реальные заявки.

## 3. Технический долг

Главный долг проекта:
- бот ещё не запущен как рабочий канал приёма заявок;
- AI ещё не подключён в backend как отдельный сервис;
- форма на public landing пока frontend-only;
- калькулятор пока живёт в Google Sheets;
- нет единого потока данных:
  public form -> backend -> bot -> manager/admin -> AI summary -> calculator/lead context.

## 4. Целевая схема

Базовая схема:

public landing / calculator
-> POST /api/public/leads
-> backend lead service
-> Telegram / MAX bot notification
-> optional AI summary
-> manager receives structured lead
-> future admin panel lead record

Важный принцип:
AI не должен блокировать заявку.
Если AI недоступен, заявка всё равно должна уйти в бот.

## 5. Рекомендуемый порядок этапов

### Этап 1 — BOT-LEADS-1A: public lead endpoint

Цель:
создать backend endpoint для приёма заявки с публичной формы.

Endpoint:
POST /api/public/leads

Payload:
{
  "name": "string",
  "phone": "string",
  "objectType": "string",
  "area": "string",
  "packageType": "string",
  "contactMethod": "string",
  "comment": "string"
}

Минимальные требования:
- endpoint public;
- не требует auth;
- валидирует длину строк;
- не принимает огромные payload;
- возвращает `{ "ok": true }`;
- сначала можно не сохранять в БД;
- не ломает существующие auth/session/cookie/API contracts.

### Этап 2 — BOT-LEADS-1B: отправка заявки в Telegram / MAX

Цель:
после приёма формы backend отправляет структурированное сообщение владельцу проекта.

Формат сообщения:

Новая заявка Danko BuildTech

Имя: ...
Контакт: ...
Способ связи: ...
Тип объекта: ...
Площадь: ...
Формат: ...
Комментарий: ...
Источник: public landing

Нужно решить:
- где хранить TELEGRAM_BOT_TOKEN;
- где хранить TELEGRAM_LEADS_CHAT_ID;
- как передавать env на Render;
- что делать при ошибке отправки;
- нужен ли MAX на первом этапе или сначала Telegram.

### Этап 3 — PUBLIC-LEADS-1A: подключить форму к backend

Цель:
заменить frontend-only submit в `useLeadFormDraft` на реальную отправку.

Требования:
- добавить состояния idle/submitting/success/error;
- не использовать приватные admin network helpers, если они завязаны на auth/session;
- public landing не должен импортировать admin controller/admin screens;
- при успехе показать:
  `Заявка отправлена. Мы свяжемся с вами выбранным способом.`
- при ошибке показать:
  `Не удалось отправить заявку. Попробуйте ещё раз или напишите нам напрямую.`

### Этап 4 — AI-BACKEND-1A: минимальный AI service

Цель:
создать отдельный backend service/use-case для AI-обработки заявки.

Первый use-case:
- принять lead data;
- структурировать комментарий;
- сформировать краткий summary для Telegram.

Пример AI summary:

Клиент хочет ремонт апартаментов 52 м².
Интересуется пакетом C.
Предпочитает связь через Telegram.
Предварительные вопросы:
1. Есть ли планировка?
2. Нужна ли мебель и техника?
3. Есть ли сроки заезда/сдачи?

Ограничения:
- не смешивать AI вызовы с роутером;
- config через env;
- graceful fallback при ошибке AI;
- не блокировать отправку заявки;
- не отправлять персональные данные в AI без отдельного решения;
- сначала можно отправлять только обезличенный текст/summary.

### Этап 5 — CALCULATOR-MIGRATION-0: аудит Google Sheets-калькулятора

Цель:
не переносить Google Sheets хаос как есть, а описать доменную модель калькулятора.

Источник:
Google Sheet `Горизонт событий D`, вкладка `Аналитика`.

Уже известные данные:
- пакеты A / B / C;
- расчёт на 63,9 м²;
- цены:
  - C: 40 228 ₽/м²;
  - B: 52 280 ₽/м²;
  - A: 75 416 ₽/м²;
- категории:
  - работы;
  - материалы;
  - мебель;
  - техника;
  - логистика и техрасходы;
  - уборка;
  - двери;
  - доп.работы;
- объекты:
  - RE — 126,0 м² — A;
  - K8 — 45,0 м² — B;
  - TK — 69,0 м² — B;
  - ИБ / 22 — 52,0 м² — C;
  - АГ / 82 — 41,5 м² — C;
  - ИБ / 42 — 47,1 м² — C.

Результат этапа:
- описать package model;
- описать categories model;
- описать formula model;
- определить, что сначала можно hardcode package model в коде;
- Google Sheets не использовать как runtime source на первом этапе.

### Этап 6 — PUBLIC-CALCULATOR-1A: первый публичный калькулятор

Цель:
сделать минимальный публичный калькулятор на сайте.

Минимальная версия:
- пользователь вводит площадь;
- выбирает пакет C/B/A;
- получает ориентир:
  - цена за м²;
  - общая сумма;
  - разбивка по категориям;
- CTA:
  - заполнить заявку;
  - перейти к форме с предзаполненными area/packageType.

Ограничения:
- сначала можно считать на frontend;
- не сохранять;
- не отправлять;
- позже связать с backend lead endpoint.

## 6. Что временно не является приоритетом

До закрытия bot/AI/calculator направления не тратить основной фокус на:
- официальный домен;
- финальную SEO-полировку;
- сложную QR-механику;
- красивую цифровую визитку;
- расширение лендинга ради красоты;
- дополнительные анимации;
- публичную регистрацию.

Эти задачи вернутся позже, когда будет готов канал заявок и расчётный контур.

## 7. Архитектурные ограничения

Сохранять:
- `/` — public landing;
- `/app` — admin UI;
- public layer не должен импортировать admin controller, admin screens или приватные network helpers;
- backend auth/session/cookie/database не менять без отдельной задачи;
- package files не менять без необходимости;
- каждый этап делать маленьким PR.

## 8. Рекомендуемый ближайший следующий шаг

Следующий рабочий этап после этого документа:

BOT-LEADS-1A

Но перед реализацией:
- провести аудит текущей backend-структуры;
- найти подходящее место для public leads route;
- проверить текущие patterns для schemas/services/errors;
- сделать маленький PR без Telegram и без AI;
- сначала только endpoint + validation + safe response.
