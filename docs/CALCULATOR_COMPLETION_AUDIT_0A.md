# CALCULATOR_COMPLETION_AUDIT_0A

## 1. Цель

Подготовить перенос комплектации кухни, мебели и техники в публичный калькулятор `/estimate`.

Этот аудит фиксирует только источник логики и границы будущего frontend-only этапа. Реализация калькулятора комплектации, изменение UI, backend, API и миграций в эту задачу не входят.

## 2. Источник

Источник аудита: `04. КП от «22» мая 2026 г. по ТЗ ЦП 1 обои.xlsx`.

Файл `.xlsx` не коммитится в GitHub и используется только как локальный источник аудита. Google Sheets / xlsx не должен становиться runtime dependency публичного калькулятора.

## 4. Кухня

Найденный блок:

- лист: `Кухня`;
- строка: 5;
- позиция: `Кухонный гарнитур прямой`;
- единица: `м.п.`;
- количество: `5.0`;
- материалы / изготовление, база: `50 400 ₽/м.п.`;
- работы / монтаж, база: `5 600 ₽/м.п.`;
- коэффициент: `1.3`;
- материалы / изготовление с коэффициентом: `327 600 ₽`;
- работы / монтаж с коэффициентом: `36 400 ₽`;
- итог: `364 000 ₽`.

В `КП смета`, строка 320, этот же раздел отражён как:

- количество: `5 м.п.`;
- материалы / изготовление: `65 520 ₽/м.п.`, всего `327 600 ₽`;
- работы / монтаж: `7 280 ₽/м.п.`, всего `36 400 ₽`;
- итог: `364 000 ₽`.

Базовый состав кухни:

- прямая кухня ориентировочно 5 м.п.;
- верхние и нижние шкафы;
- корпус ЛДСП;
- фасады ЛДСП;
- столешница Egger или аналог;
- фурнитура с доводчиками;
- подготовка / интеграция под встроенный холодильник, индукционную панель, встроенную СВЧ и духовой шкаф;
- сенсорная подсветка.

Зависимость от погонных метров есть: в источнике расчёт построен как `5 м.п. × ставка за м.п. × коэффициент`. Для первого публичного среза безопасно взять дефолт 5 м.п. и дать возможность включить / выключить кухню. Ручной ввод длины кухни можно добавить позже отдельным PR, чтобы не расширять первую реализацию.

В public 1A брать:

- `includeKitchen`;
- fixed default `kitchenLengthMeters = 5`;
- `manufacturingPricePerMeter = 65 520`;
- `installationPricePerMeter = 7 280`;
- итог `364 000 ₽`;
- строки сметы: `Кухонный гарнитур прямой`, `Монтаж кухонного гарнитура`.

Отложить:

- сложную конфигурацию кухни;
- внутреннее наполнение модулей;
- премиальную фурнитуру;
- каменную столешницу;
- нестандартные фасады;
- отдельную детализацию каждого кухонного модуля;
- кухонную технику как часть кухонного гарнитура.

## 7. Компонентная модель для public calculator

В публичном калькуляторе нельзя переносить комплектацию как один общий блок `Мебель`. Для клиента это выглядит непрозрачно: кухня, гардеробная, пеналы, мебель санузла и техника имеют разную природу, разные точки принятия решения и разные правила уточнения после замера.

Публичная модель должна разделять компоненты:

1. Кухня базовая.
2. Пенал под технику.
3. Пенал под холодильник / высокий модуль.
4. Гардеробная / шкаф-купе.
5. Мебель / пеналы санузла.
6. Техника.

Кухня, гардеробная, пеналы и техника должны включаться отдельно. Блок `Мебель` из таблицы нельзя переносить в public как одну строку.

Внутренние КП поставщиков можно использовать как источник себестоимости, состава и структуры, но public calculator должен считать продажную стоимость. В публичном frontend нельзя выводить contractor cost, себестоимость, маржу и внутреннюю закупку.

Сравнение с КП Попереко В.А. от 13.03.2026:

| Компонент | Стоимость |
| --- | ---: |
| Кухонный гарнитур | 266 800 ₽ |
| Встроенная мебель для гардеробной | 104 500 ₽ |
| Встроенная мебель в санузел | 81 900 ₽ |
| Итого | 453 200 ₽ |

В стоимость КП Попереко включены сборка, монтаж, доставка, разгрузка / погрузка и упаковка.

Вывод: КП Попереко показывает, что мебельная комплектация должна быть не одним блоком, а отдельными public options:

- `kitchenBase`;
- `kitchenAppliancePenal`;
- `kitchenFridgePenal`;
- `wardrobe`;
- `bathroomFurniture`;
- `appliances`.

Risk note:

- ставка `Мебель 361 900 ₽` из локального КП слишком агрегированная;
- перед реализацией её нельзя использовать как единую public строку;
- нужно либо разложить её по поставщикам / позициям, либо использовать только как контрольный общий ориентир.

## 8. Предлагаемый первый public-срез

Рекомендуемая модель для `PUBLIC-ESTIMATE-COMPLETION-1A`:

- `includeKitchenBase: boolean`;
- `kitchenLengthMeters: number`;
- `includeKitchenAppliancePenal: boolean`;
- `includeKitchenFridgePenal: boolean`;
- `includeWardrobe: boolean`;
- `includeBathroomFurniture: boolean`;
- `includeAppliances: boolean`;

Рекомендуемые дефолты:

- `includeKitchenBase = true`;
- `kitchenLengthMeters = 5`;
- `includeKitchenAppliancePenal = true`;
- `includeKitchenFridgePenal = true`;
- `includeWardrobe = false`;
- `includeBathroomFurniture = true`;
- `includeAppliances = true`;

Рекомендуемые ставки / totals:

| Группа | Public option | Материалы / оборудование | Работы / монтаж | Итого |
| --- | --- | ---: | ---: | ---: |
| Кухня базовая | `includeKitchenBase` | 327 600 ₽ | 36 400 ₽ | 364 000 ₽ |
| Пенал под технику | `includeKitchenAppliancePenal` | требует разложения | включено в ставку, если источник КП Попереко | требует отдельной ставки |
| Пенал под холодильник / высокий модуль | `includeKitchenFridgePenal` | требует разложения | включено в ставку, если источник КП Попереко | требует отдельной ставки |
| Гардеробная / шкаф-купе | `includeWardrobe` | 104 500 ₽ по КП Попереко | включено | 104 500 ₽ |
| Мебель / пеналы санузла | `includeBathroomFurniture` | 81 900 ₽ по КП Попереко | включено | 81 900 ₽ |
| Техника | `includeAppliances` | 245 520 ₽ | 18 600 ₽ | 264 120 ₽ |

Рекомендуемые `EstimateLineItem` категории:

- кухня: `materials` для изготовления / поставки, `works` для монтажа;
- гардеробная / шкаф-купе: `materials` или `equipment` для встроенной мебели, без отдельной работы, если ставка взята из КП Попереко;
- мебель / пеналы санузла: `materials` или `equipment` для встроенной мебели санузла, без отдельной работы, если ставка взята из КП Попереко;
- техника: `equipment` для техники, `works` для установки / подключения.

`includeDeliveryAssembly` для мебели на первом шаге не нужен как отдельная надбавка, если ставка взята из КП Попереко, потому что там доставка / монтаж уже включены. Если в будущем ставка будет без монтажа / доставки, логистику нужно считать отдельной опцией.

Предлагаемый `sectionId`: `extras` или отдельный `completion`, если будет расширяться `EstimateSectionId`. Так как текущая модель уже содержит `extras`, первый PR можно сделать без расширения типов. Если потребуется отдельная видимость в UI как `Комплектация`, можно минимально добавить `completion` отдельным PR.

## 9. Guardrails

- Первый implementation PR должен быть frontend-only.
- Не менять backend.
- Не менять `/api/public/leads`.
- Не менять database / migrations.
- Не подключать Google Sheets / xlsx как runtime source.
- Не коммитить `.xlsx`.
- Не менять существующие расчётные модули работ без отдельной задачи.
- Не смешивать комплектацию с уже реализованными разделами ремонта.
- Не делать item-level редактор кухни, мебели и техники в первом PR.
- Сохранять текущую публичную модель: расчёт предварительный, точная комплектация уточняется после замера и согласования.
