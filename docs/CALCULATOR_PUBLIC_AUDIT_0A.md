# CALCULATOR-PUBLIC-AUDIT-0A — аудит публичного калькулятора ремонта

Дата фиксации: 2026-05-24

## 1. Цель

Зафиксировать продуктовую и архитектурную модель публичного калькулятора ремонта перед реализацией.

Калькулятор должен стать отдельной публичной страницей сайта:

```text
/estimate
```

Это клиентский калькулятор для обычных пользователей, которые хотят быстро понять ориентир стоимости ремонта и передать вводные в заявку.

Это не админский калькулятор, не инструмент мастера и не перенос всего Excel в интерфейс.

## 2. Текущая рабочая точка

Основной runtime уже стабилен:

- public landing работает на `/`;
- privacy page работает на `/privacy`;
- admin app работает на `/app`;
- public lead form отправляет `POST /api/public/leads`;
- `personalDataConsent=true` обязателен;
- honeypot `website` работает;
- rate limit работает;
- Telegram delivery через public bot работает;
- public leads сохраняются в БД в отдельную таблицу `public_leads`;
- public response остаётся `{ "ok": true }`;
- admin UI для public leads сейчас не делается.

Следующий фокус: публичный калькулятор ремонта на `/estimate`.

## 3. Source material boundary

Локальный `.xlsx` используется только как источник аудита логики.

Правила:

- не добавлять `.xlsx` в git;
- не коммитить исходную таблицу;
- не подключать Google Sheets или `.xlsx` как runtime dependency;
- не делать публичный сайт зависимым от Google Sheets на первом этапе;
- переносить в репозиторий только утверждённую доменную модель, markdown-документы и код, созданный по результатам аудита.

Исходная таблица нужна для понимания:

- структуры листа `Расчет`;
- формул;
- зависимостей между блоками;
- справочников;
- состава клиентских параметров;
- итоговой модели сметы.

## 4. Product decision

Публичный калькулятор:

- живёт на `/estimate`;
- открывается с кнопки `Рассчитать стоимость`;
- предназначен для клиента, а не для администратора;
- не должен выглядеть как таблица;
- должен идти как понятная страница с последовательными блоками;
- должен сохранить логику и компактность листа `Расчет`;
- должен скрыть внутреннюю сложность Excel;
- должен давать ориентир стоимости, а не юридически окончательную смету.

Кнопки `Рассчитать стоимость` на public landing позже должны вести на `/estimate`.

## 5. What moves into public calculator

В публичный калькулятор попадает только клиентская часть:

- параметры объекта;
- помещения;
- площадь;
- высота;
- стены;
- полы;
- тёплый пол;
- потолки и свет;
- электрика;
- сантехника;
- двери;
- итоговая смета;
- отправка результата в заявку.

Не переносить в публичную версию:

- весь Excel;
- внутренние справочники как редактируемые таблицы;
- мастерские настройки;
- служебные листы;
- полную детализацию закупок;
- внутренние коэффициенты без пользовательского смысла;
- admin-only инструменты;
- runtime-зависимость от Google Sheets.

## 7. Geometry simplification

Для публичного сайта геометрию упрощаем.

Пользователь вводит площади помещений по БТИ. Он не вводит длины стен.

Базовая модель:

```text
периметр = 4 × √S × коэффициент формы
```

Где:

- `S` — площадь помещения;
- `коэффициент формы` — поправка на вытянутость/сложность помещения.

Площадь стен:

```text
площадь стен = периметр × высота
```

Проёмы считаются упрощённо:

- двери;
- окна;
- типовые вычеты;
- без ручного моделирования каждой стены.

Цель этой модели — дать разумный ориентир клиенту, не заставляя его измерять весь объект как сметчика.

## 8. Calculator model boundaries

Публичный калькулятор должен иметь доменную модель, отделённую от UI.

Рекомендуемые уровни:

- input model: что вводит пользователь;
- normalized model: очищенные и приведённые данные;
- calculation engine: детерминированные расчёты;
- result model: итог, категории, строки сметы;
- lead payload adapter: передача результата в заявку.

Engine не должен зависеть от React-компонентов.

На первом этапе можно начать с frontend-only deterministic engine, если он не требует backend-state и не хранит данные. Позже engine можно вынести в backend или разделить типы, если появится необходимость.

## 9. Output model

Публичный результат должен показывать:

- ориентир общей стоимости;
- стоимость по ключевым категориям;
- базовую расшифровку;
- выбранные параметры;
- предупреждение, что точная смета зависит от состояния объекта, выбранных материалов, мебели, техники и состава работ.

Не показывать клиенту:

- внутренние служебные коэффициенты без пояснения;
- необработанные формулы;
- raw Excel-структуру;
- внутренние закупочные поля;
- админские статусы.

## 10. Lead integration

Связь с заявкой должна появиться позже отдельным этапом.

Будущий flow:

```text
/estimate
-> calculation result
-> lead form with prefilled area/package/comment
-> POST /api/public/leads
-> public_leads
-> Telegram notification
```

На этапе калькулятора не менять текущий contract `POST /api/public/leads` без отдельной задачи.

## 12. Runtime guardrails

Сохранять:

- `/` — public landing;
- `/privacy` — privacy page;
- `/app` — admin app;
- `/api/public/leads` — текущий public lead endpoint;
- public response shape `{ "ok": true }`;
- `public_leads` отдельно от `request_drafts`;
- admin calculator отдельно от public calculator;
- Google Sheets / `.xlsx` только как audit source.

Не делать в рамках audit/first shell:

- не добавлять `.xlsx` в git;
- не подключать Google Sheets runtime;
- не менять backend;
- не менять migrations;
- не менять admin UI;
- не менять auth/session/cookie;
- не добавлять AI;
- не создавать финальный calculator engine без утверждённой модели;
- не смешивать публичный калькулятор с существующим admin calculator.
