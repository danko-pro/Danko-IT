## A. Архитектурный принцип

Утверждённый принцип модели сантехники:

1. **БАЗА атомарных позиций.** Каждая инженерная строка хранится **отдельно** и независимо:
   - `id` — стабильный машинный идентификатор (slug);
   - **публичное название** — бытовой язык для клиента;
   - **техническое название** — внутренний инженерный ярлык;
   - **категория** — `works | materials | equipment | consumables`;
   - **единица** — `шт | м.п. | м² | комплект`;
   - **rate (publicPrice)** — публичная цена позиции (= Итого/ед);
   - **комментарий** — пояснение состава;
   - **источник** — `Сан v1 (Excel) | code | manual | требует уточнения`.

2. **СЦЕНАРИЙ — агрегатор.** Сценарий не хранит собственную цену, он **ссылается** на атомарные позиции и **суммирует** их:
   - `scenarioId`, `publicTitle`, `publicDescription`;
   - `includedItems` — список ссылок на `id` атомарных позиций;
   - `defaultQuantities` — количества по умолчанию;
   - `dependencies`, `conflicts`, `warnings`.
   - **Цена сценария = Σ (rate атомарной позиции × quantity).** Сценарий сам по себе цену не задаёт.

3. **Public UI.** Сценарии показываются клиенту **бытовым языком**; спецификация раскрывает атомарный состав; **итог суммируется по 4 категориям** (`works`, `materials`, `equipment`, `consumables`) и в `total`.

4. **Терминологический контракт.** Разрешено только: `publicPrice`, `basePublicPrice`, `packageMultiplier`, `packageLevel`, `quantity`, `total`, `rate`.
   **Запрещено:** `contractorCost`, `supplierCost`, `internalCost`, «себестоимость», «закупка», «маржа», `margin`, `markup`.

5. **Недостающие позиции.** Если позиции нет в Сан v1 (например, насос/принудительная канализация) — помечаем статусом «нет в Сан v1; нужна ставка от пользователя». **Ставки не придумываем и не переносим как «из Excel».**

6. **Только frontend.** Модель и расчёт остаются клиентскими (как сейчас в `public-estimate-*.ts`). Backend/API не требуются.

## B. Схема данных (TypeScript-набросок модели)

> ⚠️ Это **иллюстрация модели в документе, а не файл в `src/`**. Не копировать в код без отдельного PR и без согласованных ставок.

```ts
// Категории затрат — соответствуют существующему EstimateCostCategory
type PlumbingCategory = "works" | "materials" | "equipment" | "consumables";

// Единицы измерения
type PlumbingUnit = "шт" | "м.п." | "м²" | "комплект";

// Источник данных по ставке
type PlumbingRateSource = "Сан v1 (Excel)" | "code" | "manual" | "требует уточнения";

// АТОМАРНАЯ ПОЗИЦИЯ — минимальная неделимая инженерная строка
type PlumbingAtomicItem = {
  id: string;                       // стабильный id, напр. "vanity-sink-set"
  publicTitle: string;              // бытовое название для клиента
  technicalTitle: string;           // внутренний инженерный ярлык
  category: PlumbingCategory;       // доминирующая категория (см. примечание ниже)
  unit: PlumbingUnit;               // единица измерения
  rate: number | null;              // publicPrice = Итого/ед; null = ставки пока нет
  // Разбивка публичной цены по 4 категориям (сумма даёт rate):
  breakdown: {
    works: number | null;
    materials: number | null;
    equipment: number | null;
    consumables: number | null;
  };
  source: PlumbingRateSource;       // "Сан v1 (Excel)" | code | manual | требует уточнения
  comment?: string;                 // пояснение состава
  // Инженерный учёт точек (для подсчёта узла/кранов), как в текущем коде:
  coldWaterPoints?: number;
  hotWaterPoints?: number;
  sewerPoints?: number;
};

// СЦЕНАРИЙ — агрегатор ссылок на атомарные позиции, без собственной цены
type PlumbingScenario = {
  scenarioId: string;
  publicTitle: string;              // бытовой заголовок
  publicDescription: string;        // что входит, простыми словами
  includedItems: Array<{
    itemId: string;                 // ссылка на PlumbingAtomicItem.id
    defaultQuantity: number;        // количество по умолчанию
  }>;
  dependencies?: string[];          // id сценариев/позиций, без которых нельзя
  conflicts?: string[];             // взаимоисключающие сценарии/позиции
  warnings?: string[];              // предупреждения для менеджера/клиента
  // total НЕ хранится: считается как Σ(item.rate × quantity)
};
```

**Примечание по полю `category`.** В Сан v1 (как и в коде) у каждой ставки заданы сразу 4 под-значения (`works/materials/equipment/consumables`). Поэтому атомарная позиция хранит **разбивку `breakdown` по 4 категориям** в одной строке, а `category` отражает доминирующую категорию для группировки в UI. Маппинг колонок Excel: **Работа → `works`, Материал → `materials`, Расходники → `consumables`, Оборудование → `equipment`; `rate` = Итого/ед = сумма четырёх.**

**Маппинг колонок:** числа в Сан v1 идут в порядке `works | materials | consumables | equipment`; в таблицах ниже сохранён тот же порядок. `rate` = Итого/ед.

- Атомарная база `plumbingRates` дополнена 9 позициями из Сан v1, нужными для P0-сценариев: `extraWaterPoint` (8 500), `extraSewerPoint` (8 000), `toiletFanOutlet` (3 800), `showerDrain` (36 000), `showerPartition` (24 000), `concealedShowerMixer` (52 500), `tiledShowerTray` (34 500), `showerEnclosure` (40 000), `showerStackMixer` (16 500). Существующие 16 ставок не менялись.
- Введена сценарная модель (агрегатор-ссылка на атомарные позиции, цена = Σ позиций × qty).
- Реализованы **два** сценария: «Перенос унитаза / инсталляции» (ориентир 20 300 ₽) и «Душевая зона» (вариант «поддон из плитки» = 147 000 ₽, вариант «душевой уголок» = 92 500 ₽). Состав показывается через окно спецификации `EstimateSpecOverlay`.
- Сценарии выключены по умолчанию; «Душевая зона» взаимоисключается с «Ванной» (`includeBath`).

### H.2. Следующие инкременты (НЕ в этом PR)

- **Демонтаж / монтаж стен мокрой зоны** — отдельный будущий модуль (нет в Сан v1: `ph-wet-zone-wall-demolition`, `ph-wet-zone-wall-build`).
- **Насос / принудительная канализация** — отдельная будущая фича. **Нет в Сан v1**, ставку не выдумываем; до её получения только предупреждение клиента (используется в «Переносе инсталляции» и «Постирочной с насосом»).
- **Труба по метражу (перенос инсталляции, инкремент).** Модель длины: коэффициент по площади санузла — при **СУ ≤ 5 м²** факт переноса трубы ~**2.5–3 м**; при **СУ > 5 м²** — закладывать **5 м.п. на санузел**. **ОЖИДАЕТ ставку ₽/м.п.** с разбивкой `works / materials / consumables` от пользователя; до получения ставка **не добавляется** (в Сан v1 погонной ставки нет — `ph-pipe-routing`, `ph-sewer-pipe-110`). Будет подключено в сценарий «Перенос инсталляции» инкрементом.
- **Прочие сценарии** (Кухня с ПММ, Расширенный узел, Постирочная, Санузел стандарт) — все ставки уже есть в Сан v1, но собираются в следующих PR.

> Принцип сохраняется: **ставки не выдумываем** (особенно труба и насос) — они помечены 🟡 и ждут данных от пользователя.
